1. Introduction

Background of the Project

In computer science education, sorting algorithms represent fundamental building blocks taught to introduce runtime complexity analysis, divide-and-conquer methodologies, and data manipulation. However, traditional teaching methods—such as static diagrams, pseudo-code trace slides, or passive animation loops—frequently fail to engage students effectively. Many struggle to bridge the gap between abstract algorithmic principles and actual step-by-step array index manipulations. Sort Pulse was conceived to address this gap by transforming theoretical sorting operations into an interactive, time-attack 2D arcade puzzle game. By gamifying manual selection, bubble swaps, shifts, and partitions, the application builds deep physical intuition and developer muscle memory regarding the operational flow of five classic sorting algorithms.

Purpose of the System

The primary purpose of Sort Pulse is to serve as an interactive desktop learning simulator. Players do not just observe sorting; they manually execute the step-by-step logic. The system validates every action against a pre-computed theoretical algorithm trace. Correct selections trigger point rewards, combo multipliers, and timing extensions, while mistakes trigger instant score and time penalties. This tight, real-time feedback loop forces players to actively inspect array values and construct sorting paths, accelerating the absorption of theoretical sort logic.

Project Objectives

  • Interactive Mechanics: Design five customized gameplay mechanisms modeling the operational flow of Selection, Bubble, Insertion, Quick (Lomuto), and Merge Sort.
  • Visual and Audio Feedback: Build a custom graphics engine handling smooth block slide animations, vector particle bursts, and a runtime synth audio sequencer that speeds up under time pressure.
  • OOP Architecture: Apply core Object-Oriented principles—Encapsulation, Abstraction, Inheritance, and Polymorphism—to structure a clean, maintainable MVC codebase.
  • Persistent Tracking: Maintain top local score entries in a flat-file database using automated serialization.

2. System Design and Overview

Description of the Program

Sort Pulse is a standalone desktop application written in Java and rendered on a JavaFX 2D Canvas. It runs at 60 FPS via a native AnimationTimer thread loop. Players can choose between Timed Blitz Mode (intense puzzle solving against a clock) and Practice Mode (untimed learning environment). The game features dual visuals (Classic Neon and retro GameBoy style), dynamic wave scaling (array sizes growing from 8 to 12 blocks as scores stack), and an 8-bit synthesizer running on a dedicated audio mixer thread.

Scope and Limitations

The system is bounded by specific scope constraints and limitations:

  • Scope: Supports ascending sort order across five modes; synthesizes custom frequencies on-the-fly; persists high scores locally; allows fullscreen toggles and theme configurations.
  • Limitations: Restricted to local file storage (no cloud database); supports a maximum array length of 12 (to prevent screen clutter and maintain quick play loops); handles only single-player profiles.

Overview of the Data Structure(s) Used

The application relies on several core data structures to model the game state and drive calculations:

  • Polymorphic Arrays (BlockSegment[]): The grid is structured as an array of the abstract class BlockSegment. This array is managed by a wrapper class PuzzleRow, which implements an array-bounds safety catch stack.
  • Generic Lists (java.util.List<SortingStep>): Used to pre-calculate and store the sequence of correct index positions and comparison parameters. The controller checks player input against the active index in this list.
  • Entity Lists: Parallel ArrayLists store active visual nodes (e.g., Particle vectors, FloatingText tags) updated dynamically in the frame loop.
  • Local Flat-File (sort_pulse_scores.txt): Acts as a local serialization database. Leaderboard scores are parsed and written as delimited comma-separated records.

Overview of the Algorithm Implemented

The system operates on two algorithmic levels: back-end tracing and real-time validation. When a wave spawns, a theoretical sorting tracer (such as bubble sort or quick sort Lomuto partition) runs on a copy of the array. It records every index swap, pivot assignment, and subarray boundary into a step queue. During play, the controller monitors keyboard inputs, compares the selected index to the step queue, and moves the grid elements only if the player's action is valid.

Real-World Application or Use Case

Sort Pulse is directly applicable as an educational classroom module for secondary or university-level data structure courses. It also serves as an excellent training tool for software developers preparing for technical interviews, turning dry theoretical questions into a fast-paced reflex trainer.

3. Program Design

Data Structures Design

The design features the wrapper class PuzzleRow, which encapsules the primitive array of BlockSegment objects. The PuzzleRow class contains a secondary overflowStack array. If visual coordinates mismatch or index bounds are exceeded due to external synchronization delays, the system intercepts the error, copies the segment to the safety stack, logs the warning, and avoids a game-terminating crash.

Algorithm Design

The user interaction flows are customized for each algorithm mode:

  • Selection Sort: Select minimum element in active range -> press ENTER to shift to sorted position.
  • Bubble Sort: Traverses array -> identify adjacent unsorted elements -> press ENTER to swap.
  • Insertion Sort: Highlight active unsorted element -> press ENTER repeatedly to shift it leftward.
  • Quick Sort: Rightmost element is pivot -> compare current block -> if smaller, press ENTER to move to target slot -> finally select pivot to lock.
  • Merge Sort: Compare heads of Subarray A and B -> select smaller -> press ENTER to merge down.

Application of Object-Oriented Programming principles

Encapsulation: The internal variables representing the game's core metrics, such as gameState, score, and countdownTimer, are encapsulated in ChromaCascadeModel as private properties. They can only be accessed or mutated through explicit getter and setter methods, ensuring state integrity.

Abstraction: An abstract class BlockSegment serves as the template for all grid block items. It hides the details of sort weights behind an abstract signature calculateSortWeight(), allowing the controller to manipulate grid rows without needing to know specific implementation details.

Inheritance: Inheritance is visible where OddSegment and EvenSegment derive from the abstract base class BlockSegment, inheriting structural traits while overriding weight logic. Similarly, ChromaCascadeApp inherits from JavaFX's Application class.

Polymorphism: Polymorphism is utilized when rendering the grid. The view treats all items in the row array as BlockSegment objects. When drawing labels, calling segment.calculateSortWeight() invokes the subclass-specific calculation dynamically at runtime.

4. Program Implementation

Description of Major Classes and Methods

  • ChromaCascadeApp.start(Stage stage): Builds the JavaFX application window, creates the root StackPane layout container, instantiates the MVC components, loads the pixel-art font resources, and shows the primary stage.
  • ChromaCascadeModel.getTheme(): Returns the colors of the active theme (Classic Neon or GameBoy Retro).
  • ChromaCascadeView.draw(): Clears the screen and draws the boundaries, blocks, status logs, HUD text, particles, and floating banners onto the active canvas.
  • ChromaCascadeController.handleKeyPress(KeyCode code): Listens to keyboard inputs. Moves the cursor left or right on A or D, and triggers actions on ENTER.
  • GridSorter.compare(BlockSegment a, BlockSegment b): Compares the raw values of two blocks, placing empty elements (nulls) at the end.
  • SoundManager.mixerThread: A background thread that mixes active tones and writes raw 16-bit stereo PCM bytes to the output buffer to play procedural audio.
  • LeaderboardManager.addScore(String algo, int score, String initials): Reads the leaderboard file, inserts the new entry in sorted order if it qualifies for the top 5, and writes the updated list back to the disk.

Explanation of Program Workflow

The operational workflow follows these structured phases:

  1. Initialization Phase: The app loads. It reads javafx_path.txt, registers visual layouts, and displays the main menu. The leaderboard data is read from the local file into memory.
  2. Configuration Phase: The player selects a theme, enables or disables Practice Mode, and clicks an algorithm button. This triggers the initialization sequence, generating a random array of blocks.
  3. Tracing Phase: Before gameplay starts, GridSorter runs the selected algorithm on a copy of the array and records the correct comparisons and swaps in a list of SortingStep objects.
  4. Active Game Loop Phase: An AnimationTimer triggers at 60 FPS. The controller checks keyboard inputs against the active SortingStep. Correct inputs execute swaps and chimes; incorrect inputs trigger penalties and buzzes. The background thread generates a synth bass arpeggio that scales tempo with the clock.
  5. Wave Clear Phase: Once all steps are completed, a wave clear banner displays. The system awards points, adds time, increases array length, and generates a new random wave.
  6. Game Over & Save Phase: When the timer hits zero, the controller stops the music. If the score qualifies for the leaderboard, the game prompts the player for initials, updates the score file, and returns to the menu.

1. Dynamic Coordinate Interpolation

To achieve smooth sliding animations, block coordinates are updated in the view using a linear interpolation formula. Rather than snapping immediately to target grid cells, coordinates are adjusted incrementally by moving a fraction (22%) of the remaining distance to the target position each frame:

// Interpolate block rendering coordinates toward target coordinates
double targetX = startX + i * boxWidth;
double targetY = startY;

if (targetAlgo.equalsIgnoreCase("Merge Sort")) {
    // Map blocks to upper/lower levels depending on merge state
    if (model.getSteps() != null && model.getCurrentStep() < model.getSteps().size()) {
        SortingStep step = model.getSteps().get(model.getCurrentStep());
        if (i < step.mergeTarget) {
            targetY = startY + 60; // Lower output level
        } else {
            targetY = startY - 60; // Upper unmerged level
        }
    }
}

double currentX = visualXMap.getOrDefault(segment, targetX);
double currentY = visualYMap.getOrDefault(segment, targetY);

// Interpolate coordinates at 22% velocity per frame
currentX += (targetX - currentX) * 0.22;
currentY += (targetY - currentY) * 0.22;

visualXMap.put(segment, currentX);
visualYMap.put(segment, currentY);

3. Lomuto Quick Sort Partition Step Tracer

The system pre-computes sorting steps to validate player actions in real-time. In Lomuto partitioning, a loop compares elements against the pivot and records each comparison and swap in the step list:

private static int partitionTrace(BlockSegment[] array, int low, int high, List<SortingStep> steps) {
    BlockSegment pivot = array[high];
    int i = low - 1;
    for (int j = low; j < high; j++) {
        // Record comparison step
        SortingStep comp = new SortingStep();
        comp.activeLeft = low; comp.activeRight = high; comp.pivotIndex = high; comp.nextCursor = j;
        steps.add(comp);

        if (array[j].getRawValue() <= pivot.getRawValue()) {
            i++;
            // Record swap step
            SortingStep swap = new SortingStep();
            swap.correctIndex = j; swap.targetIndex = i;
            swap.activeLeft = low; swap.activeRight = high; swap.pivotIndex = high;
            steps.add(swap);

            BlockSegment temp = array[i]; array[i] = array[j]; array[j] = temp;
        }
    }
    return i + 1;
}

5. Algorithm Analysis

Complexity Overview

The time and space complexities of the five implemented algorithms are summarized below:

Algorithm Best Case Average Case Worst Case Space Complexity Stability
Selection Sort Θ(n2) Θ(n2) Θ(n2) O(1) Unstable
Bubble Sort Θ(n) Θ(n2) Θ(n2) O(1) Stable
Insertion Sort Θ(n) Θ(n2) Θ(n2) O(1) Stable
Quick Sort Θ(n log n) Θ(n log n) Θ(n2) O(log n) Unstable
Merge Sort Θ(n log n) Θ(n log n) Θ(n log n) O(n) Stable

Discussion of Efficiency and Performance

During interactive gameplay, these mathematical complexity profiles translate directly into physical effort. In **Bubble Sort**, players must compare and swap adjacent elements. A highly unsorted 12-element array can require dozens of swaps, making gameplay feel long and demanding. In contrast, **Selection Sort** requires exactly $n-1$ swaps across all configurations. While simple to play, it lacks early-termination shortcuts, meaning the game length remains fixed and predictable. **Insertion Sort** is highly efficient when the array is nearly sorted, requiring very few actions. However, a reverse-sorted array forces the player to shift every element all the way to the left, which can be exhausting. **Quick and Merge Sort** use divide-and-conquer strategies. While they are mathematically faster for computers, they require more complex decision-making from players. In Merge Sort, players must perform double-level head comparisons, which requires more cognitive effort than the straightforward linear scanning of Selection Sort.

Discussion of Objective Achievement

Sort Pulse successfully meets its educational goals. The time pressure in Timed Blitz Mode discourages trial-and-error play, forcing players to think ahead and execute the actual steps of each sorting algorithm. In contrast, Practice Mode provides a stress-free environment for beginners to study the algorithms at their own pace without penalties. The real-time visual and audio feedback helps players self-correct, reinforcing their understanding of the algorithms.

Observations and Findings

During testing, we observed that Bubble Sort requires the most manual cursor movement, reflecting its high number of swap operations. In contrast, Merge Sort is easy to trace visually due to its double-level layout, which clearly separates unmerged subarrays from the sorted output area. In Quick Sort, the Lomuto comparison flags make it easy for players to identify the split points and understand how elements are partitioned around the pivot.

7. Conclusion

Summary of the Project

Sort Pulse is an interactive educational game built using JavaFX and a Model-View-Controller architecture. By translating sorting algorithms into engaging gameplay mechanics, the system provides a practical platform for learning data structures. The procedural audio synthesizer and dual visual themes deliver an immersive retro arcade experience without external file dependencies.

Key Learnings and Insights

Pre-calculating sorting steps as a list of independent structures (SortingStep) decoupled the game logic from visual rendering, preventing synchronization issues. Developing the procedural synth engine also demonstrated that dynamic audio can be generated directly using standard Java audio libraries, keeping the application lightweight.

Recommendations for Future Improvements

  • Networked Leaderboards: Replace the local scoring file with a remote API server to support global leaderboards and online competition.
  • Custom Algorithm Builder: Integrate an in-game editor that allows users to write custom sorting routines and compile them into playable stages.
  • Mobile Support: Package the application with Gluon Mobile for Android and iOS devices, expanding its reach as a mobile learning tool.
